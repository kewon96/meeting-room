## 1. 회의 목적

1:1(chat.md)에서 선택지가 ①풀 self-host(Mimir+Loki+Tempo) vs ②AMP 하이브리드(AMP+Loki+Tempo)로 좁혀지고, 범위가 "전용 관측 계정 + cross-account 토폴로지"로 확대되며 토론 모드로 전환. 4개 안건을 검토.

## 3. 종합 판단 (주치의)

세 입력이 한 점으로 수렴한다.

- **안건 1의 함의**: AMP exemplar 미지원은 ①(Mimir)의 유일하게 단단한 명분이다. **그러나** 그것이 영향을 주는 건 metric→trace 점프 **한 곳뿐**이고, ⓐ Tempo↔Loki 의 trace↔log 상관은 그대로 살아 있으며, ⓑ AMG data-link 매크로로 2급 우회가 되고, ⓒ exemplar 는 dev 에서 **가장 덜 쓰이는** 상관이다(critic 의 YAGNI). → exemplar 갭은 **dev 에서 Mimir self-host 를 정당화하기에 약하다.**
- **안건 2·3의 합의**: dev 에서 self-host 메트릭 백엔드는 요금·운영·cross-account·가용성 어느 축에서도 AMP 대비 우위가 없다. 사용자 본인의 "Mimir만 먼저" 직감이 옳았다.

→ **dev 의 메트릭은 AMP 로 간다. exemplar 갭은 수용한다.** metric→trace 점프가 dev 에서 실측으로 아쉬워지는 것이 ①(Mimir) 재고의 트리거이지, 선제적 self-host 의 근거가 아니다.

## 4. 결정 (Decisions)

| # | 결정 | 비고 |
|---|------|------|
| D1 | **선택지 ③(X-Ray) 탈락 확정** | X-Ray native 상관 파트너가 CloudWatch Logs → "Grafana correlation" 기조 위반 |
| D2 | **dev 메트릭 백엔드 = AMP** (Mimir self-host 보류) | exemplar 미지원은 수용. lock-in 낮음(remote_write/PromQL 표준) |
| D3 | **단계화 채택 — Phase 1 은 같은 계정·메트릭만** | 전용 관측 계정 격리는 Phase 3(스테이징/프로덕션 승격)로 연기 |
| D4 | **exemplar(metric→trace) 갭 수용**, Tempo↔Loki 상관으로 충분 | dev 에서 metric→trace 실측 수요 발생 시 ① 재평가 |

### 단계화 경로 (확정)

- **Phase 1 (지금, dev)**: `/actuator/prometheus` → Alloy(또는 agent) → **AMP** → **AMG**. **같은 계정**, cross-account 0, self-host 0. 메트릭만.
- **Phase 2 (로그가 실측으로 아쉬울 때)**: 같은 계정에 `svc_observability` 서비스 + **Loki** 추가, Cloud Map 내부 DNS 연결. Tempo 는 trace 점프를 실제로 쓰는 게 관찰될 때까지 보류.
- **Phase 3 (스테이징/프로덕션 승격 트리거)**: **전용 관측 계정 격리 + cross-account 배선**(peering/TGW/PrivateLink + IAM). dev 에서 검증 끝 → "이사"지 "신축" 아님.

### 전환 게이트

- **P1→P2**: dev 에서 "메트릭만으로 못 푼 디버깅 + 로그 상관이 있었으면 빨랐다"가 **실측**으로 N회.
- **P2→P3**: 스테이징/프로덕션 승격이 로드맵 확정, 또는 관측 대상이 2개 이상으로 증가.
- **②→① 재평가**: active series 폭증으로 AMP 월요금이 Mimir compute+S3 추정 추월, 또는 metric→trace 점프가 dev 핵심 워크플로우가 됨.

## 5-1. 결정 갱신 (2026-06-28 회의 직후, 1:1 외래)

**사용자(의사결정권자)가 D3 단계화를 부분 번복.** dev 부터 **cross-account 로 직행**한다. 같은 계정에서 시작 후 이관하는 경로(Phase 1 in-account)는 거부.

- **근거 1**: 이관 비용 > 초기 cross-account 구축 비용. dev 에서 굳힌 토폴로지를 나중에 계정 넘겨 옮기는 게 더 큰일.
- **근거 2**: log account 가 모니터링 전용이 아님 — 로그 관련 기능을 추가 구현 예정 → 전용계정 ROI 가 모니터링 하나에 갇히지 않음 → critic 의 "보호 대상이 dev 하나뿐" 전제 무효화.

**갱신된 결정**:
- ~~D3 (Phase 1 = 같은 계정, cross-account 0)~~ → **번복**: Phase 1 부터 cross-account.
- D2(메트릭=AMP) **유지** — 오히려 cross-account 에서 AMP 가 IAM/SigV4 라 VPC 면제 → 안건 2 analyst 가 짚은 "3→2 신호" 이점이 실제로 발현되는 시나리오가 됨.
- D1(③ 탈락), D4(exemplar 갭 수용) 유지.
- **남은 미정**: ⓐ 첫 신호 범위(메트릭만 vs 메트릭+로그) ⓑ cross-account 연결 방식(peering/TGW/PrivateLink) ⓒ AMP·AMG·Loki 를 각각 어느 계정에 둘지. → 다음 설계 대상.
